- **2026-06-19** — Adding a NEW required field to a persisted document contract breaks reading old rows: `run_traces.trace` is stored JSON, so making `RunStats.cost_usd` `.nullable()` (key required) throws in `RunTrace.parse` on traces written before the field existed. ## Codebase Patterns

- **2026-06-27** — `code_chunks` is NOT populated by the repo-intel indexer (grep: the only writer is the conventions seed). So the conventions extractor reads file content from the LOCAL GIT CLONE for any synced repo — `GitClient.listFiles` (new; `git ls-files`, honors .gitignore) + `git.readFile` — and falls back to `code_chunks` only for the un-cloned demo repo (seeded via `seed-conventions.ts`). A "synced" repo has `repos.clone_path` set; `repoIntel.getConventionSamples` needs `file_rank` (full index) which a merely-cloned repo lacks, so sampling falls back to a shallow-path heuristic. Trust boundary: the model returns a short DISTINCTIVE ANCHOR (rule name / identifier / config key) — NOT a multi-line snippet, because models paraphrase code (quotes, `;`, whitespace) so verbatim snippet-matching is flaky (a vue-starter-kit re-scan went 3→0). `verifyCandidates` locates the anchor (exact, then ws-normalized substring) and extracts the REAL surrounding lines as the snippet; anchor-not-found → dropped. This took verification from ~40% to ~95% of proposed candidates surviving. Evidence: `src/modules/conventions/{content,verify,extract}.ts`, `src/adapters/git/simple-git.ts`.
- **2026-06-26** — The Skills feature shipped MOST of its plumbing in the Part-0 starter BEFORE the lesson: the `skills`/`skill_versions`/`agent_skills` tables (`db/schema/skills.ts`, `agents.ts`), the `Skill`/`SkillType`/`SkillSource`/`AgentSkillLink` contracts, the agents-side link methods (`agentsRepo.linkedSkills/setSkills/linkSkill`), AND reviewer-core's prompt support (`PromptParts.skills` → `## Skills / rules` block). Only the **skills CRUD module + run-executor wiring + UI** were missing. Lesson: grep the schema/contracts before assuming a feature needs new tables. Evidence: `src/modules/skills/`, `src/db/schema/skills.ts`.
- **2026-06-26** — Wiring skills into a review run is one fetch in `run-executor.ts`: `this.agents.linkedSkills(agent.id)` → `.filter(l => l.skill.enabled).map(l => l.skill.body)` → pass as `skills` to `reviewPullRequest`. reviewer-core then populates `outcome.assembly.skills`, so the trace's skills block AND the token delta come for free. A DISABLED skill stays linked but is omitted — that omission is exactly the control-experiment lever (skill on vs off). Evidence: `src/modules/reviews/run-executor.ts` (~`:191`).
- **2026-06-19** — `@devdigest/shared` contracts are vendored as TWO hand-maintained copies — `server/src/vendor/shared/` and `client/src/vendor/shared/` — resolved by tsconfig path alias, NOT auto-synced. Adding a field (e.g. `cost_usd`) means editing both in lock-step; the only intended diffs are comments. Evidence: `server/src/vendor/shared/contracts/trace.ts`, `platform.ts`.
- **2026-06-19** — `completeAgentRun`'s `values` shape is declared in TWO places that must match: the repo fn (`src/modules/reviews/repository/run.repo.ts`) AND the interface wrapper (`src/modules/reviews/repository.ts`). Adding a field (e.g. `costUsd`) to one without the other fails typecheck.
- **2026-06-19** — PR-list per-PR cost is computed ON READ in `GET /repos/:id/pulls`, not denormalized: leftJoin `agent_runs` on the latest review's `run_id` and surface that run's `cost_usd` (null until reviewed). Evidence: `src/modules/pulls/routes.ts`.
- **2026-06-19** — `RunTrace.stats` is assembled in TWO spots in `run-executor.ts` — the success path (~`:265`) AND the error/fallback trace (~`:425`). A new stats field (e.g. `cost_usd`) must be set in both. Evidence: `src/modules/reviews/run-executor.ts`.
- **2026-06-19** — (supersedes the leftJoin-latest-run note above) PR-list COST is now the SUM of ALL of a PR's `agent_runs.cost_usd` — `sql\`sum(${agentRuns.costUsd})\`` grouped by `pr_id` — i.e. "total spend to review this PR", not the latest run's cost. Per-run cost still shows in the timeline/trace. Null (no priced run) → "—". Evidence: `src/modules/pulls/routes.ts`.
- **2026-06-19** — PR-list FINDINGS column data is computed on read in `GET /repos/:id/pulls`: take each PR's latest review, load its findings, and use `rollupSeverities` (already in `pulls/status.ts`) for the `findings_critical/warning/suggestion` counts + a capped (`FINDINGS_PREVIEW_CAP`) `findings` preview embedded in `PrMeta` for the column's hover tooltip. Findings-table `severity`/`category`/`kind` are `text`, so cast to the `Finding` enums when mapping. Evidence: `src/modules/pulls/routes.ts`.
- **2026-06-19** — (supersedes the note above) PR-list FINDINGS now unions the LATEST review PER AGENT (not just the single newest review), so every agent that ran is represented while a re-run of an agent supersedes its older review — no stale/duplicate findings from the same agent. Dismissed findings are excluded (`isNull(findings.dismissedAt)`). Dedup logic is the pure, unit-tested `latestReviewsPerAgent(reviewsNewestFirst)` in `status.ts` (key = `agentId ?? runId ?? id`). NOT handled (deliberately, learning system): cross-commit staleness (no per-review SHA in schema) and cross-agent semantic dedup. SCORE column still = the single newest review. Evidence: `src/modules/pulls/routes.ts`, `src/modules/pulls/status.ts`.

## Tool & Library Notes

- **2026-06-27** — The cheap extraction model (`openrouter/deepseek-v4-flash`) intermittently replies in CHINESE for an open-ended "extract rules" prompt — the system prompt MUST say "Respond in ENGLISH only" or half the candidates come back non-English. Also: a structured `completeStructured` call is non-deterministic even at `temperature:0` (one run proposed 4 / verified 0 with paraphrased snippets; the next proposed 5 / verified 5 verbatim) — so verification must tolerate minor reformatting (normalized-substring fallback) and the UI should show proposed-vs-verified counts. Evidence: `src/modules/conventions/extract.ts`.
- **2026-06-26** — The Skill editor's per-version "what changed" message needs a `message` column on `skill_versions` that the Part-0 schema did NOT have — added it (`db/schema/skills.ts` → `pnpm db:generate` → migration `0011_dapper_piledriver.sql` = `ALTER TABLE "skill_versions" ADD COLUMN "message" text` → `pnpm db:migrate` against the running Postgres). It's threaded only on a body change: `SkillsRepository.update(patch.versionMessage)` → `snapshotVersion(row, version, message)`. (The upstream lesson-2 branch added the same column as `0011_far_stryfe.sql` — same shape, different migration name.) Evidence: `src/modules/skills/repository.ts`, `src/db/migrations/0011_dapper_piledriver.sql`.
- **2026-06-26** — Skill import needs NO new deps: the upload is base64-in-JSON (`{filename, content_base64}`), not multipart, and the zip is read with a dependency-free reader over `node:zlib` (`src/modules/skills/unzip.ts`) that walks the central directory and decompresses ONLY the markdown core — executable/binary entries are listed but never inflated or run. Two gotchas baked in: (1) `Buffer.from(str,'base64')` NEVER throws (it silently drops bad chars) — validate the decoded length instead of try/catching it; (2) `inflateRawSync` needs `{ maxOutputLength }` + an `uncompressedSize` pre-check or a tiny archive is a heap-OOM zip bomb. Evidence: `src/modules/skills/{unzip,import,routes}.ts`.
- **2026-06-19** — New DB column flow: edit `db/schema/*.ts` → `pnpm db:generate` (drizzle-kit emits `00NN_*.sql`) → `pnpm db:migrate` (NOT applied on boot). If Docker is down and you hand-author the SQL, you MUST also add the matching entry to `db/migrations/meta/_journal.json` or drizzle won't track it. Evidence: `src/db/migrations/0010_add_cost_usd_to_agent_runs.sql`, `meta/_journal.json`.
